# 경기 입력 재대결 제한 안내 설계

## 목표

경기 결과 입력에서 첫 선수를 선택하면 도전 범위 안의 상대를 계속 표시하되, 동일 상대와의 2주 재대결 제한이 남은 선수는 선택할 수 없게 하고 다시 경기할 수 있는 날짜를 즉시 안내한다.

## 사용자 동작

- 어느 쪽 선수를 먼저 선택해도 같은 규칙을 적용한다.
- 반대쪽 선택 목록은 기존처럼 활성 선수 순서 기준 위·아래 `challengeRange`명만 표시한다.
- 범위 안에 있지만 재대결 제한 중인 상대는 목록에서 숨기지 않는다.
- 제한 중인 상대는 비활성화하고 `순위 · 이름 · M월 D일부터 가능`으로 표시한다.
- 직전 경기일로부터 `rematchCooldownDays`일째 되는 날부터 다시 선택할 수 있다. 예를 들어 7월 1일 경기와 14일 제한이면 7월 15일부터 가능하다.
- 제출 순간에는 기존 Supabase 트랜잭션이 같은 규칙을 다시 검증한다. 화면 검증은 안내와 실수 방지용이며 최종 권한은 서버에 있다.

## 데이터 흐름

`getSupabaseMatchValidationContext`는 현재 시즌 경기만 `previousMatches`로 반환해 Postgres의 `record_public_match` 규칙과 일치시킨다. `GET /api/clubs/[club]/matches`는 현재 서울 날짜를 기준으로 아직 제한이 남은 조합만 다음 형태로 정제한다.

```ts
type RematchCooldown = {
  playerAId: string;
  playerBId: string;
  availableOn: string;
};
```

API 응답은 기존 `players`, `challengeRange`에 `rematchCooldowns`를 추가한다. 같은 조합의 기록이 여러 개면 가장 늦은 경기의 제한 종료일 하나만 반환한다. 활동 선수 선택에 쓰이지 않는 조합과 이미 종료된 제한은 반환하지 않는다.

클라이언트는 첫 선수와 상대 후보의 ID 조합으로 `rematchCooldowns`를 조회한다. 제한 정보가 있으면 `<option disabled>`로 렌더링하고 `availableOn`을 한국어 월·일로 표시한다.

## 규칙 경계

- 현재 시즌의 확정 경기만 적용한다.
- 날짜는 `YYYY-MM-DD` 단위로 계산하며 서울 날짜를 오늘로 사용한다.
- 정확히 제한 종료일인 조합은 차단하지 않는다.
- 부상 및 탈퇴 선수는 기존과 동일하게 선택 목록에서 제외한다.
- DB 스키마와 RLS는 변경하지 않는다.

## 오류 처리

- 옵션 API 조회 실패는 기존 오류 메시지 흐름을 유지한다.
- 오래 열린 화면에서 제한 상태가 바뀌어도 POST의 Supabase 트랜잭션이 최종 거절하고 기존 서버 메시지를 보여준다.
- 잘못된 날짜 데이터는 DB의 `date` 타입을 신뢰하며 별도 복구 규칙을 만들지 않는다.

## 테스트

- 조회 컨텍스트가 과거 시즌 경기를 제외하는지 검증한다.
- 재대결 가능일 계산과 정확히 제한 종료일에 허용되는 경계를 검증한다.
- GET 응답이 진행 중인 제한만, 최신 조합 하나로 반환하는지 검증한다.
- 첫 번째 또는 두 번째 선수를 먼저 골라도 제한 상대가 비활성화되는지 검증한다.
- 비활성화된 옵션에 다시 가능한 날짜가 표시되는지 검증한다.
- 기존 도전 범위 필터링과 정상 제출 흐름이 유지되는지 전체 테스트로 확인한다.
